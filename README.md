## 📌 스터디 컨벤션 (Naming)
모든 명명 규칙은 **`이름-날짜-문제번호`** 흐름으로 통일합니다.
| 구분 | 형식 | 예시 |
| :--- | :--- | :--- |
| **디렉토리** | `이름/주차/문제번호/` | `조의재/1주차/boj12865/` |
| **브랜치** | `이름-날짜-문제번호` | `mokjak-260203-boj12865` |
| **커밋** | `feat: 이름 날짜 문제번호` | `feat: 조의재 260203 boj12865` |
| **PR 제목** | `[이름] 날짜 문제번호 문제명` | `[조의재] 260203 BOJ 12865 평범한 배낭` |

## 📂 디렉토리 구조
모든 작업은 개인 성함 폴더 내에서 진행합니다.

```text
 [이름]/
    └── [주차]/
        └── [문제]/
            ├── Main.java
            └── README.md (문제 설명 및 풀이 전략)

```
EX) `조의재/1주차/boj12865/Main.java`

## 🛠 스터디 프로세스

### 1. 문제 풀기 전 pull & 브랜치 생성
항상 main 브랜치를 최신 상태로 만든 뒤, **새 브랜치**를 생성합니다. (날짜는 YYMMDD 6자리)
```bash
# main 브랜치로 이동 후 최신 코드 가져오기
git checkout main
git pull origin main

# 새 브랜치 만들기 및 이동 (백준:boj, SWEA:s)
git checkout -b 이름-날짜-문제번호
```
Ex) `mokjak-260203-boj12865`


### 2. 코드 완성 후 push와 PR

```bash
# 변경 사항 추가 및 커밋
git add .
git commit -m "feat: 이름 날짜 문제번호"

# 원격 저장소에 내 브랜치 올리기
git push origin 브랜치명

```
**PR 생성 시 다음 설정을 반드시 수행해 주세요:**
1. **Assignee**: 본인 선택 (담당자 표시)
2. **Projects**: `algorithm-fairy-02` 선택 (현황 추적)

### 3. 코드 리뷰 및 머지(Merge)
* **리뷰:** 동료의 코드를 확인하고 의견을 남기거나, 문제가 없다면 **Approve**를 누릅니다.(하루 최소 2개 이상)
* **머지 조건:** **최소 2명 이상의 Approve**가 쌓이면 작성자가 직접 **Merge pull request**를 누릅니다.
* **브랜치 삭제**: 머지가 완료된 브랜치는 로컬에서 삭제하여 목록을 관리합니다.(권장)
  * `git branch -d 브랜치명` (로컬 브랜치 삭제 명령어)

## ⚠️ 주의사항
* Force Push 금지: `git push -f`는 다른 사람의 코드를 지울 수 있으므로 절대 사용하지 않습니다.
* 폴더 주의: 본인의 폴더 외에 다른 사람의 폴더나 파일을 수정하지 않도록 주의해 주세요.
